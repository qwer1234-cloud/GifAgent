## 1. 系统概览

GifAgent 是一个运行在本地的影视片段智能管理工具。它能：

-   扫描你收藏的 GIF/图片/视频目录，自动去重
-   调用本地大模型分析每一帧的**审美特征**和**情感表达**
-   生成结构化的影视标签（构图、光影、色调、情绪等）
-   建立向量索引，支持"以图搜图"和相似推荐
-   提供 Gradio 可视化界面，快速审核和反馈

### 核心设计原则

```
视觉模型(llava:13b) → 看懂画面、识别审美
文本模型(9B LLM)   → 整理标签、归纳偏好
Embedding 模型     → 相似度与聚类
SQLite + FAISS     → 记忆与检索
Gradio UI          → 人工反馈闭环
```

### 数据流

```
E:\data\originals\
  → 文件扫描 + SHA256/pHash 去重
  → GIF 抽帧 (ffmpeg, 每张 6-12 帧)
  → llava:13b 逐帧审美分析
  → 9B LLM 综合标注
  → Ollama Embedding 向量化
  → FAISS 索引
  → 偏好评分
  → Gradio 审核
  → 反馈写入数据库
```

## 2. 环境要求

| 组件 | 要求 | 说明 |
|------|------|------|
| Python | 3.11+ | uv 自动下载管理 |
| uv | 最新版 | `powershell -Command "irm https://astral.sh/uv/install.ps1 \| iex"` |
| ffmpeg | 任意版本 | 需在 PATH 中，用于 GIF 抽帧和视频处理 |
| Ollama | 最新版 | 运行在 WSL 或本机，监听 `localhost:11434` |


## 4. Library Workbench 使用指南

### 4.1 概述

Library Workbench 是 GifAgent 的统一管理界面，包含 7 个标签页：

| 标签页 | 功能 |
|--------|------|
| 今日 (Today) | 注意收件箱：任务失败、媒体冲突、画像发布、高价值候选、冠军配置晋级 |
| 队列 (Queue) | 任务引擎控制：启动/暂停/恢复/取消批量处理 |
| 审核 (Review) | 候选 GIF 审核：分页画廊，Like/Dislike/Skip/Favorite/Quality Reject |
| 搜索 (Search) | 语义搜索 + 过滤：全文、标签、文件夹、时长、状态、日期 |
| 合集 (Collections) | 智能合集：生成、刷新、冻结、导出（JSON + PBF 双格式）|
| 实验室 (Lab) | 质量实验室：基准运行、盲测 A/B、冠军晋级/回滚 |
| 设置 (Settings) | 配置编辑器 + 画像管理 + 发布控制 |

### 4.2 今日 (Today Tab)

显示所有需要注意的项目：
- 任务引擎失败（job 级 = error，stage 级 = warning）
- 媒体 SHA256 冲突（同一文件出现多次）
- 偏好画像发布事件
- 高分待审核候选（`final_score >= 0.7` 的 `candidate`）
- 冠军配置晋级事件

每条项目包含操作按钮，点击即可跳转到详情页面。

### 4.3 队列 (Queue Tab)

批量视频处理的控制面板：
- 列出所有 job 及其状态统计（pending/running/success/failed/needs_attention）
- 查看每个 job 的详细视频列表和阶段进度
- 下发控制命令：cancel / pause / resume
- 支持旧版队列 UI（设置 `GIFAGENT_LEGACY_QUEUE_UI=1` 回退）

### 4.4 审核 (Review Tab)

候选 GIF 审核界面：
- 先选择数据根目录 → 点击 `Load Folders` → 选择具体文件夹
- 分页画廊（每页 12 个缩略图），点击缩略图在右侧查看完整 GIF
- 6 种反馈：Like / Dislike / Neutral / Skip / Quality Reject / Favorite
- 键盘快捷键：A=Like, S=Neutral, D=Dislike, E=Edit Tags
- 仅当前页面的 GIF 被加载，不会一次性拉取全部数据

### 4.5 搜索 (Search Tab)

语义 + 过滤搜索：
- 输入文字进行全文本搜索（支持 FTS5 分词和词干提取）
- 可选过滤条件：标签、文件夹、时长范围、状态、日期范围
- 结果分页显示（每页最多 24 项），显示摘要和预览缩略图
- 搜索结果可直接用于创建智能合集

### 4.6 合集 (Collections Tab)

智能合集管理：
- **创建合集**：指定名称、搜索条件、目标数量、多样性权重
- **刷新合集**：执行搜索 + 最远优先多样性选择，生成新版本
- **冻结合集**：锁定当前版本，防止意外刷新
- **导出合集**：生成 JSON manifest + 二进制 PBF 文件，适合 PotPlayer 书签

导出文件位于 `data/exports/` 目录：
- `collection_{id}_v{version}.json`：可读的 JSON 清单
- `collection_{id}_v{version}.pbf`：二进制书签文件

### 4.7 实验室 (Lab Tab)

质量实验室界面：
- 查看实验运行和指标记分卡
- 创建和参与盲测 A/B 会话
- 晋级冠军配置 / 回滚到上一版本
- 查看晋级历史

### 4.8 设置 (Settings Tab)

配置和画像管理：
- 在线编辑 `configs/models.yaml` 配置
- 刷新、选择、发布偏好画像
- 查看当前生效画像版本和历史发布记录

## 5. 配置文件

所有配置位于 `configs/models.yaml`：

```yaml
# === 媒体处理 ===
media:
  source_dir: "E:/data/originals"   # 媒体源目录
  image_max_side: 1344              # 图片最长边限制
  gif_sample_frames: 8              # GIF 最少抽帧数
  gif_max_sample_frames: 12         # GIF 最多抽帧数
  video_sample_fps: 1               # 视频每秒抽帧数

# === VLM 视觉模型 ===
vlm:
  provider: "ollama"
  model: "llava:13b"                # 也可以换成 minicpm-v:8b 等
  base_url: "http://localhost:11434"
  batch_size: 50                    # 每批次处理帧数
  max_batch_size: 100

# === LLM 文本模型 ===
llm:
  provider: "ollama"
  model: "fredrezones55/Qwen3.5-Uncensored-HauhauCS-Aggressive:9b"
  base_url: "http://localhost:11434"
  require_json: true

# === 模型调度 ===
scheduler:
  model_switch_wait: 10             # 模型切换后等待秒数
  max_retries: 3                    # 切换失败最大重试

# === 偏好评分权重 ===
scoring:
  visual_similarity_weight: 0.50    # 视觉相似度
  text_similarity_weight: 0.25      # 文本标签相似度
  emotional_match_weight: 0.15      # 情感匹配度
  cluster_confidence_weight: 0.10   # 聚类置信度
  high_match_threshold: 0.80        # ≥0.80 标记为 high_match
  review_threshold: 0.60            # ≥0.60 标记为 maybe

# === 去重 ===
dedup:
  phash_hamming_threshold: 5        # pHash 汉明距离阈值

# === 存储路径 ===
database:
  path: "data/library.db"
paths:
  frames_dir: "data/frames"
  thumbs_dir: "data/thumbs"
  faiss_dir: "data/faiss"
  exports_dir: "data/exports"
```

### 通过环境变量覆盖配置

```bash
set GIFAGENT_CONFIG=configs/production.yaml
uv run uvicorn app.main:app
```

### 更换模型

编辑 `configs/models.yaml` 中 `vlm.model` 和 `llm.model` 字段即可。例如：

```yaml
vlm:
  model: "minicpm-v:8b"    # 更省显存，可与 9B 共存

llm:
  model: "qwen3.6:35b-a3b" # 更强的文本推理
```

更换后重启服务即可生效，无需改代码。

## 6. 完整工作流

### 6.1 通过 API 逐步执行

```bash
# 确保 FastAPI 已启动: uv run uvicorn app.main:app --host 127.0.0.1 --port 8000
```

#### 阶段 1: 扫描入库

```bash
curl -X POST http://127.0.0.1:8000/api/scan
```

**返回示例：**
```json
{
  "status": "ok",
  "stats": {
    "scanned": 9278,
    "registered": 3500,
    "skipped_sha256": 120,
    "skipped_phash": 45,
    "skipped_ext": 5613
  }
}
```

这一步会：
-   递归扫描 `E:/data/originals/`
-   计算每个文件的 SHA256 和感知哈希
-   自动跳过完全重复和近似重复的文件
-   从文件名提取电影名（如 `The Dark Knight_xxx.gif` → `The Dark Knight`）
-   写入 `media` 表

#### 阶段 2: 抽帧

```bash
curl -X POST http://127.0.0.1:8000/api/preprocess
```

**返回示例：**
```json
{
  "status": "ok",
  "stats": {
    "total": 3500,
    "processed": 3480,
    "frames_extracted": 27840,
    "failed": 20
  }
}
```

这一步对每个 GIF 执行：

```bash
ffmpeg -i input.gif -vf "fps=2,scale=640:-1" data/frames/media_xxx_frame_%06d.jpg
```

每张 GIF 生成 6-12 张 640px 宽的 JPEG 帧，写入 `frames` 表。

#### 阶段 3: VLM 分析 + LLM 整理（核心，耗时最长）

```bash
curl -X POST http://127.0.0.1:8000/api/process-frames
```

**这是最耗时的步骤。** 系统会：

1.  停止 9B 文本模型，加载 llava:13b
2.  批量分析 50-100 帧 → 每帧生成 `caption`、`emotional_core`、`aesthetic_notes`、`why_i_like_it`
3.  停止 llava:13b，加载 9B 文本模型
4.  按 media_id 分组，综合所有帧分析 → 生成 GIF 级别的 `summary`、`tags`、`scene_type`

**预估耗时：** 对于 9000 张 GIF（约 7 万帧），大约需要 8-20 小时，取决于帧复杂度和 Ollama 推理速度。

**监控进度：**

```bash
curl http://127.0.0.1:8000/api/processing-progress
```

返回：
```json
{
  "total": 72000,
  "done": 15000,
  "failed": 3,
  "pending": 56997
}
```

#### 阶段 4: 构建 FAISS 索引

```bash
curl -X POST http://127.0.0.1:8000/api/build-index
```

对每个已标注的 media 计算 embedding，写入 FAISS 索引。

#### 阶段 5: 偏好评分

```bash
curl -X POST http://127.0.0.1:8000/api/score-all
```

对每个 GIF 计算四因子加权偏好分，输出决策等级。

### 6.2 一键全流程

```bash
uv run python scripts/index_library.py
```

等价于依次执行阶段 1-5，适合首次全量索引。

### 6.3 查看总览

```bash
curl http://127.0.0.1:8000/api/status
```

```json
{
  "media_count": 3547,
  "frame_count": 28376,
  "frames_pending": 0,
  "annotated_media": 3500,
  "indexed_vectors": 3500
}
```

## 7. API 参考

所有 API 基础路径：`http://127.0.0.1:8000`

### 7.1 状态与进度

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/status` | 总览统计 |
| GET | `/api/processing-progress` | VLM 处理进度 |

### 7.2 媒体导入

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/scan` | 扫描源目录，注册新文件 |
| POST | `/api/preprocess` | 对未处理的 GIF 抽帧 |

### 7.3 AI 处理

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/process-frames` | 启动 VLM+LLM 处理（后台线程） |
| POST | `/api/build-index` | 构建 FAISS 向量索引 |
| POST | `/api/score-all` | 批量偏好评分 |

### 7.4 查询与反馈

| 方法 | 路径 | 参数 | 说明 |
|------|------|------|------|
| GET | `/api/media/{media_id}` | - | 获取媒体详情、标注、帧列表 |
| GET | `/api/media/{media_id}/score` | - | 获取单个媒体的偏好评分 |
| POST | `/api/feedback` | `media_id`, `rating`, `tags`, `reason` | 提交人工反馈 |

**反馈提交示例：**

```bash
curl -X POST "http://127.0.0.1:8000/api/feedback?media_id=media_abc123&rating=like&tags=The+Dark+Knight,Joker,名场面&reason=光影对比太强了"
```

## 10. 数据库说明

### 10.1 表结构

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| media | 媒体文件元数据 | media_id, file_path, sha256, phash, film |
| frames | GIF 抽帧记录 | frame_id, media_id, frame_path, vlm_status |
| annotations | 媒体综合标注 | annotation_id, summary, emotional_core, tags_json, scene_type |
| frame_annotations | 逐帧分析 | annotation_id, caption, aesthetic_notes_json, why_i_like_it |
| feedback | 人工反馈 | feedback_id, user_rating (like/dislike/neutral), corrected_tags_json |
| vector_refs | 向量索引映射 | vector_id, owner_type, owner_id, vector_type |
| video_clips | 视频候选片段 | clip_id, start, end, status (candidate/approved/rejected/exported) |

### 10.2 vlm_status 状态机

```
pending → vlm_processing → text_inferring → done
                                            → failed
```

### 10.3 查询示例

```bash
# 查看所有已标注的 GIF
uv run python -c "
from app.db import get_connection
conn = get_connection()
rows = conn.execute('''
  SELECT m.media_id, m.film, a.emotional_core, a.scene_type
  FROM media m JOIN annotations a ON m.media_id=a.media_id
  LIMIT 10
''').fetchall()
for r in rows:
    print(f'{r[\"film\"]:30s} | {r[\"emotional_core\"]:15s} | {r[\"scene_type\"]}')
"

# 按情绪统计分布
uv run python -c "
from app.db import get_connection
conn = get_connection()
rows = conn.execute('''
  SELECT emotional_core, COUNT(*) as cnt
  FROM annotations GROUP BY emotional_core ORDER BY cnt DESC
''').fetchall()
for r in rows:
    print(f'{r[\"emotional_core\"]:20s} {r[\"cnt\"]:5d}')
"
```

## 11. 常见问题

### Q: Ollama 跑在 WSL 里，Python 在 Windows 上，能通吗？

**能。** Ollama 监听 `0.0.0.0:11434`，WSL 自动转发到 Windows 的 `localhost:11434`。所有 API 调用走 HTTP，不依赖文件系统共享。确保 WSL 里：

```bash
# 在 WSL 中确认 Ollama 监听地址
ollama serve
# 默认监听 127.0.0.1:11434，需要改为 0.0.0.0
# 设置环境变量: OLLAMA_HOST=0.0.0.0:11434
```

或通过 Windows 端 Ollama 直接连接（如果安装了 Windows 版）。

### Q: 显存不够怎么办？

16GB 显存只能交替运行 llava:13b 和 9B，这是 `ModelScheduler` 的设计前提。如果显存更小：

-   换用 `minicpm-v:8b` (~5.5GB) 可与 9B 共存
-   或降低量化等级到 Q2/Q3

### Q: 处理太慢怎么办？

-   调整 `vlm.batch_size` 增大批次
-   调低 `media.gif_sample_frames` 减少抽帧数
-   `media.image_max_side` 降低图片分辨率
-   先跑一批小样本验证 pipeline 正确，再全量跑（睡觉时挂机）

### Q: 标签质量不好怎么办？

1.  检查 `configs/prompts.yaml` 里的 prompt 是否适合你的内容类型
2.  尝试更换 VLM 模型
3.  通过 Gradio UI 积累 feedback 后，可以训练个性化 ranker（第三阶段功能）

### Q: 数据库文件能迁移吗？

**能。** `data/library.db` 是标准 SQLite 文件，`data/faiss/` 下是 FAISS 二进制文件。直接拷贝到新机器即可，注意更新 `file_path`（目前存的是绝对路径）。

### Q: Gradio 版本冲突怎么办？

项目用 `uv` 管理依赖，使用 `.venv` 隔离。如果还是遇到冲突：

```bash
uv sync --reinstall
```
