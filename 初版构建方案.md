# GifAgent 初版构建方案（修订版）

## 0. 修订说明

本版本基于苏格拉底式追问后修订，与原方案的主要差异：

| 维度 | 原方案假设 | 修正后 |
|------|-----------|--------|
| 内容类型 | 动漫/混合 | **纯真人影视** |
| 标注重点 | gif_worthiness, 聊天可用性 | **审美分析 + 情感连接** |
| VLM 策略 | 本地模型可能够用 | **llava:13b 换入换出，质量优先** |
| 数据量 | 1000 测试目标 | **9000+ GIF（约 5-10 万帧）** |
| 视频 | 长片场景切分 | **20s 以内短视频** |
| 偏好闭环 | 多维度评分 | **MVP 只收集，不消费反馈** |
| 去重 | 分层 | **全自动（sha256 + phash）** |
| Embedding | 专用模型部署 | **Ollama 现有能力** |

---

## 1. 项目目标

构建一个本地优先的影视 GIF/视频片段自动标注与偏好挖掘 Agent。系统从用户已经保存的真人影片 GIF 和视频中学习个人审美偏好，并扩展到浏览视频自动推荐适合截取成 GIF 的经典片段。

核心定位：**影视片段收藏 — 收集喜欢的电影/剧集经典镜头，重观赏性**

初版目标：

- 个人审美偏好向量库
- 图片 / GIF 自动标注能力（审美 + 情感维度）
- 相似内容检索能力
- 低置信度人工反馈闭环（收集反馈，先不消费）
- 后续视频截 GIF 的候选片段发现能力

核心原则：

```text
视觉模型负责看懂内容和审美
文本模型负责整理标签和归纳偏好
Embedding 模型（Ollama）负责相似度与聚类
数据库负责记忆
Agent 负责调度和决策
人工反馈负责补充 badcase（MVP 只收集）
```

## 2. 当前约束与设计判断

### 2.1 数据特点

- 9000+ 真人影片 GIF，存储在 `E:\data\originals\`
- 文件名按电影名前缀组织（如 `The Dark Knight_xxx.gif`），无角色/场景关键词
- 该目录下混合少量静态图片和 20s 以内短视频
- 已有数据基本都是用户喜欢的，没有系统化 badcase

因此初版不训练监督分类器，采用 one-class preference profile：

```text
已有收藏 = 正样本偏好分布
新样本 = 判断接近哪个偏好簇，而不是直接判定 bad
```

输出分级：

- 高匹配
- 可能喜欢
- 待确认
- 未知

**不输出"不喜欢"，除非用户明确反馈。**

### 2.2 模型策略

**硬件：RTX 4070Ti Super 16GB 显存**

实际部署模型：

| 角色 | 模型 | 大小 |
|------|------|------|
| 文本推理 | `fredrezones55/Qwen3.5-Uncensored-HauhauCS-Aggressive:9b` | 6.5 GB |
| 视觉理解 | `llava:13b` (Q4) | ~7.4 GB |
| Embedding | Ollama 现有模型（nomic-embed-text 或 9B 文本模型） | - |

**llava:13b 为纯文本模型 9B 的内置视觉能力不足的解决方案。**

模型调度采用换入换出策略：

```text
ollama stop llava:13b
ollama run qwen3.5:9b → 文本推理 / 标签整理
ollama stop qwen3.5:9b
ollama run llava:13b → 批量处理 50-100 帧
ollama stop llava:13b
ollama run qwen3.5:9b → 下一批
```

换入换出代价 ~10-15s/次，批量处理摊销后影响有限。9B 和 llava:13b 不能同时加载（6.5+7.4 > 16GB）。

### 2.3 环境工具链

所有工具需要从零安装：
- ffmpeg（Windows 安装）
- Python 依赖：Pillow, watchdog, imagehash, fastapi, gradio 等
- Ollama 需拉取 `llava:13b`

## 3. 总体架构

```text
E:\data\originals\ （图片 / GIF / 视频）
  → 文件扫描（watchdog + 定时全扫）
  → sha256 去重（完全一致直接跳过）
  → phash 去重（汉明距离 < 阈值，自动跳过）
  → 提取电影名前缀（文件名 → film 字段）
  → 媒体预处理（GIF 抽 8-12 帧 / 视频抽帧）
  → 模型调度器（VLM 批量 → 文本推理 交替）
  → Ollama Embedding（文本 + 图像）
  → FAISS 索引 + SQLite 元数据
  → 偏好评分（相似度排序，不判 bad）
  → Gradio 审核界面（A/S/D/E 快捷键）
  → 反馈写入 feedback 表（MVP 只收集不消费）
  → 后续用于视频片段发现与 GIF 导出
```

推荐技术栈：

```text
语言：Python
后端：FastAPI
审核界面：Gradio
媒体处理：ffmpeg, Pillow, OpenCV
元数据：SQLite
向量库：FAISS
Embedding：Ollama 现有模型（nomic-embed-text 或 9B 模型 API）
视觉理解：llava:13b（Ollama）
LLM 编排：Ollama API（本地 OpenAI-compatible）
```

## 4. 模块设计

### 4.1 文件入口模块

职责：

- 扫描 `E:\data\originals\` 下的图片、GIF、视频
- 监听新文件（watchdog）
- 计算 sha256（完整文件 hash）
- 计算 phash（感知 hash）
- 全自动去重：sha256 完全一致 + phash 汉明距离 < 阈值 均自动跳过
- 从文件名提取电影名前缀
- 写入基础元数据

基础元数据结构：

```json
{
  "media_id": "media_000001",
  "file_path": "E:/data/originals/The Dark Knight_xxx.gif",
  "media_type": "image | gif | video",
  "film": "The Dark Knight",
  "sha256": "...",
  "phash": "...",
  "width": 640,
  "height": 480,
  "duration": 3.2,
  "frame_count": 72,
  "created_at": "2026-06-13T10:00:00+08:00",
  "indexed_at": "2026-06-13T10:10:00+08:00"
}
```

### 4.2 媒体预处理模块

职责：

- 图片缩放（最长边限制 1024-1344）
- GIF 抽关键帧（6-12 帧）
- 视频抽帧（<20s 短视频，1fps）
- 生成缩略图和预览资源
- 所有帧保留，不预去重（让 VLM 看完整动态变化）

图片处理：

```text
原图 → 统一最长边限制 1024 或 1344 → 送入 VLM / embedding
```

GIF 处理：

```text
GIF → 读取 duration / frame_count → 抽 6-12 个关键帧 → 保存为 jpg
```

视频处理（短视频简化版）：

```text
视频（<20s）→ 1fps 抽帧 → 生成候选 2-4 秒片段 → 保存 keyframes
```

推荐命令：

```bash
ffmpeg -i input.gif -vf "fps=2,scale=640:-1" frames/frame_%06d.jpg
ffmpeg -i input.mp4 -vf "fps=1,scale=320:-1" frames/frame_%06d.jpg
```

### 4.3 模型调度模块（新增）

职责：

- 管理 Ollama 模型生命周期
- 实现 VLM 批量处理 → 文本推理 的交替调度
- 跟踪每帧处理状态（pending → vlm_processing → text_inferring → done / failed）
- 记录进度（已处理数 / 总帧数，估算剩余时间）
- 处理模型切换失败的重试逻辑

调度状态机：

```text
IDLE
  → START_BATCH: ollama stop 9B, wait 10s, ollama run llava:13b
  → VLM_PROCESSING: 批量发送 50-100 帧到 llava:13b
  → BATCH_COMPLETE: ollama stop llava:13b, wait 10s, ollama run 9B
  → TEXT_INFERRING: 用 9B 整理标签、归纳偏好、输出 JSON
  → BATCH_DONE: 检查是否还有未处理帧
      → YES: 回到 START_BATCH
      → NO: 回到 IDLE
  → ERROR: 重试最多 3 次，仍失败标记为 failed
```

### 4.4 视觉理解模块

职责：

- 对图片或关键帧生成审美 caption
- 提取情感核心、审美特征、个人连接理由
- 标注维度聚焦于影视审美 + 情感

单帧输出格式：

```json
{
  "frame_index": 3,
  "timestamp": 1.5,
  "caption": "角色在暖色灯光下侧脸特写，浅景深模糊了背景",
  "emotional_core": "melancholy",
  "aesthetic_notes": ["暖色调", "浅景深", "侧光"],
  "why_i_like_it": "光影对比强化了角色的孤独感"
}
```

GIF / 视频综合输出格式：

```json
{
  "media_type": "gif",
  "film": "The Dark Knight",
  "summary": "Joker 在审讯室逐渐逼近镜头，从冷笑转为狰狞",
  "emotional_core": "tension",
  "aesthetic_notes": ["冷色调", "近距离特写", "硬光", "手持镜头感"],
  "why_i_like_it": "Heath Ledger 的微表情在 3 秒内完成了完整的情绪弧线",
  "tags": ["The Dark Knight", "Joker", "Heath Ledger", "审讯室", "名场面"],
  "scene_type": "dialogue"
}
```

### 4.5 Embedding 与向量库模块

职责：

- 使用 Ollama API 生成图像和文本向量
- 进行相似内容召回
- 聚类用户偏好
- 为视频候选片段做粗筛

初版保存三类向量：

```text
media_global_embedding：整张图片或整个 GIF 的聚合向量（帧向量均值）
frame_embedding：GIF / 视频关键帧向量
text_embedding：summary + tags + emotional_core 的文本向量
```

向量库：FAISS + SQLite（小规模下足够）

### 4.6 偏好画像模块

职责：

- 从用户收藏中学习偏好簇
- 给新图片 / GIF 计算偏好匹配分
- 判断是否需要人工审核

初版评分（MVP 只收集反馈，不消费）：

```json
{
  "similarity_to_saved_items": 0.84,
  "cluster_match": "tension_closeup",
  "final_score": 0.81,
  "decision": "high_match | maybe | review | unknown",
  "needs_review": false
}
```

final_score 简单加权：

```text
final_score =
  0.50 * visual_similarity
  + 0.25 * text_similarity
  + 0.15 * emotional_match
  + 0.10 * cluster_confidence
```

### 4.7 人工反馈模块

职责：

- 展示候选图片 / GIF / 视频片段
- 展示模型标签、推荐理由、相似历史样本
- 让用户快速确认或修正
- 将反馈写回数据库（MVP 只收集，后续阶段消费）

界面设计：

```text
左侧：当前预览
右侧：标签、summary、推荐理由、相似历史样本
底部：喜欢 / 不喜欢 / 一般 / 改标签 / 导出
快捷键：A 喜欢，S 一般，D 不喜欢，E 编辑标签
```

反馈数据结构：

```json
{
  "media_id": "media_000123",
  "user_rating": "like | dislike | neutral",
  "corrected_tags": ["The Dark Knight", "Joker"],
  "favorite_reason": "Heath Ledger 的表演层次感",
  "reviewed_at": "2026-06-13T11:00:00+08:00"
}
```

### 4.8 视频截 GIF 模块

这是第二阶段功能，但初版架构预留。由于视频源均为 <20s 短视频，无需场景切分。

流程：

```text
视频文件（<20s）
  → 1fps 抽帧
  → 对关键帧算 embedding
  → 与喜欢库做相似度召回
  → Top N 交给 VLM 精评
  → 人工预览确认
  → ffmpeg 导出 GIF / WebP / MP4
```

## 5. 数据库初版设计

```sql
CREATE TABLE media (
  media_id TEXT PRIMARY KEY,
  file_path TEXT NOT NULL,
  media_type TEXT NOT NULL,
  film TEXT,

  phash TEXT,
  width INTEGER,
  height INTEGER,

  frame_id TEXT PRIMARY KEY,
  media_id TEXT NOT NULL,

  frame_index INTEGER,
  timestamp REAL,
  width INTEGER,
  height INTEGER,
  vlm_status TEXT DEFAULT 'pending',
  FOREIGN KEY(media_id) REFERENCES media(media_id)

CREATE TABLE annotations (
  annotation_id TEXT PRIMARY KEY,
  media_id TEXT NOT NULL,
  model_name TEXT,

  why_i_like_it TEXT,
  tags_json TEXT,
  raw_json TEXT,
  created_at TEXT,
  FOREIGN KEY(media_id) REFERENCES media(media_id)

CREATE TABLE feedback (
  feedback_id TEXT PRIMARY KEY,
  media_id TEXT NOT NULL,
  user_rating TEXT,
  corrected_tags_json TEXT,
  favorite_reason TEXT,
  reviewed_at TEXT,
  FOREIGN KEY(media_id) REFERENCES media(media_id)
);

CREATE TABLE vector_refs (
  vector_id TEXT PRIMARY KEY,
  owner_type TEXT NOT NULL,
  owner_id TEXT NOT NULL,
  vector_type TEXT NOT NULL,

CREATE TABLE video_clips (
  clip_id TEXT PRIMARY KEY,

  exported_path TEXT,
  created_at TEXT,
  FOREIGN KEY(video_id) REFERENCES media(media_id)
);
```

## 7. Prompt 与模型输出约束

所有模型输出必须要求 JSON。

### 7.1 VLM 帧分析 prompt

```text
You are analyzing individual frames from a movie/TV show GIF.
Focus on the CINEMATIC and AESTHETIC qualities, not just listing objects.

Output JSON only:
{
  "caption": "concise description of the scene, composition, and what makes it visually striking",
  "emotional_core": "tension | melancholy | awe | joy | sadness | catharsis | serenity | excitement | dread | nostalgia | admiration | other",
  "aesthetic_notes": ["specific cinematic qualities: lighting, color palette, depth of field, framing, texture, movement"],
  "why_i_like_it": "a personal, subjective reason this frame is compelling - think like a cinephile"
}
```

### 7.2 文本模型综合标注 prompt

```text
You are an AI that synthesizes frame-by-frame analysis into a cohesive movie scene annotation.

Given multiple frame analyses from the same GIF, output a single JSON:
{
  "summary": "one cohesive sentence describing the full moment captured in this GIF",
  "emotional_core": "the dominant emotion carried through the scene",
  "aesthetic_notes": ["consolidated list of the most significant cinematic qualities across all frames"],
  "why_i_like_it": "an eloquent, personal reason this moment is worth saving - what makes it cinematically special",
  "tags": ["film_title", "character_name", "actor_name", "scene_type", "notable_keywords"],
  "scene_type": "close-up | dialogue | action | transition | reaction | establishing | montage | other"
}

Film title hint (from filename): {film_name}
```

### 7.3 JSON fallback 策略

```text
1. 尝试严格 JSON parse
2. 失败则提取第一个 JSON object（正则匹配 { ... }）
3. 仍失败则标记为 analysis_failed
4. 不把坏输出写入正式 annotation
```

## 8. MVP 实施里程碑

目标：
- 安装 ffmpeg、Python 依赖

- 扫描 `E:\data\originals\`

验收：
- 能导入 9000+ 个图片/GIF
- 重复文件自动跳过
- 每个 GIF 有 6-12 张关键帧
- film 字段正确提取

目标：

目标：
- 使用 Ollama API 生成图像和文本向量

验收：

目标：
- 基于相似度给出推荐等级

### 阶段 5：人工反馈界面

目标：

验收：

## 9. 初版配置

```yaml
media:
  source_dir: "E:/data/originals"
  image_max_side: 1344
  gif_sample_frames: 8
  gif_max_sample_frames: 12
  video_sample_fps: 1
  candidate_clip_min_seconds: 2.0
  candidate_clip_max_seconds: 4.0

embedding:
  provider: "ollama"
  image_model: "llava:13b"       # 或 nomic-embed-text
  text_model: "qwen3.5:9b"       # 或 nomic-embed-text

vlm:
  provider: "ollama"
  model: "llava:13b"
  batch_size: 50                  # 每批次处理帧数
  max_batch_size: 100

llm:
  provider: "ollama"
  model: "fredrezones55/Qwen3.5-Uncensored-HauhauCS-Aggressive:9b"
  require_json: true

scheduler:
  model_switch_wait: 10           # 模型切换等待秒数
  max_retries: 3

scoring:
  visual_similarity_weight: 0.50
  text_similarity_weight: 0.25
  emotional_match_weight: 0.15
  cluster_confidence_weight: 0.10
  high_match_threshold: 0.80
  review_threshold: 0.60

dedup:
  phash_hamming_threshold: 5
```

## 10. 风险与处理

### 风险 1：没有 badcase，模型过度推荐

处理：初版不输出 hard bad。设置 review 队列。用用户反馈慢慢补充 dislike/neutral（MVP 只收集）。

### 风险 2：llava:13b 换入换出效率

处理：批量 50-100 帧/次，摊销切换时间。如果切换后模型加载失败，最多重试 3 次。

### 风险 3：Ollama embedding 质量不足

处理：先跑通。如果检索效果不满意，后续切换 sentence-transformers + CLIP/bge 专用模型。

### 风险 4：标签不稳定

处理：固定 JSON schema。高频标签做字典映射。9B 模型做 tag normalization。

### 风险 5：向量相似不等于用户喜欢

处理：相似度只做召回，不做最终判断。结合情感匹配和人工反馈。后续训练个性化 ranker。

### 风险 6：llava:13b 对影视帧的审美理解

处理：通过 prompt 工程引导关注 cinematic qualities。如果效果差，考虑换 minicpm-v:8b 或其他 VLM。

### 风险 7：Windows 下 ffmpeg 路径问题

处理：配置文件显式指定 ffmpeg 路径。提供安装脚本。
